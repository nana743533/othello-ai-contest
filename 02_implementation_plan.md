# CodinGame Othello AI C++実装方針提案書

本ドキュメントでは、CodinGameのOthelloコンテストで上位入賞を目指すための、強力なオセロAIのC++実装方針を提案します。

## 1. 盤面表現：ビットボード（Bitboard）

オセロAIのパフォーマンスの要となるのが盤面表現です。本実装では**ビットボード**を採用します。

### 1.1. 概要
8x8の64マスの盤面を、64ビット整数（`uint64_t`）2つで表現します。
- `uint64_t player_board`: 自分の石の配置
- `uint64_t opponent_board`: 相手の石の配置

### 1.2. メリット
- **高速な合法手生成と石の反転**: ビットシフトと論理演算（AND, OR, XOR）のみで処理できるため、ループ処理が不要になり極めて高速です。
- **省メモリ**: 盤面全体をわずか16バイトで表現できるため、後述する置換表（Transposition Table）に大量の局面をキャッシュできます。
- **SIMD命令の活用**: 将来的な拡張として、AVX2などのSIMD命令を用いてさらに高速化することが可能です。

## 2. 探索アルゴリズム：NegaScout（PVS） + 反復深化

探索アルゴリズムには、Minimax法の発展形である**NegaScout法（Principal Variation Search）**を採用します。

### 2.1. アルゴリズムの選定理由
オセロにおいては、モンテカルロ木探索（MCTS）よりも、評価関数を用いたMinimax系の探索（Alpha-Beta法やNegaScout法）の方が強いことが経験的に知られています。NegaScout法はAlpha-Beta法をさらに最適化したもので、探索空間を大幅に削減できます。

### 2.2. 反復深化（Iterative Deepening）
CodinGameの厳しい時間制限（1ターン150ms）を最大限に活用するため、反復深化を導入します。
- 深さ1から順に探索を行い、制限時間が近づいた時点で探索を打ち切り、その時点での最善手を返します。
- これにより、時間切れによるランダムな手の選択を防ぎます。

### 2.3. 高速化テクニック
- **置換表（Transposition Table）**: Zobrist Hashingを用いて盤面のハッシュ値を計算し、探索結果（評価値、最善手、探索深さ）をキャッシュします。同一局面の再探索を防ぎます。
- **Move Ordering（手の順序付け）**: Alpha-Betaカットを効率的に発生させるため、探索する手の順番を工夫します。
  1. 置換表に保存されている前回の最善手
  2. 角を取る手
  3. 評価関数でスコアが高い手

## 3. 評価関数（Heuristic Evaluation Function）

オセロの評価関数は、ゲームの進行度（序盤・中盤・終盤）に応じて重みを動的に変化させる必要があります。

### 3.1. 評価要素
以下の4つの主要な要素を組み合わせて評価値を計算します。

| 評価要素 | 説明 | 重要度 |
| :--- | :--- | :--- |
| **Mobility（着手可能数）** | 自分が打てる合法手の数と、相手が打てる合法手の数の差。相手の選択肢を奪うことがオセロの基本戦略です。 | 序盤〜中盤で極めて重要 |
| **Corner（角の確保）** | 角（a1, h1, a8, h8）の確保状況。角は一度取られると絶対にひっくり返されないため、非常に価値が高いです。 | 常に重要 |
| **Stability（確定石）** | 今後絶対にひっくり返されない石（確定石）の数。角を起点に辺に沿って増えていきます。 | 中盤以降で重要 |
| **Coin Parity（石の数）** | 盤面上の自分の石と相手の石の数の差。 | 終盤のみ重要（序盤は無視） |

### 3.2. 進行度に応じた重み付け
ゲームの進行度（空きマスの数）に応じて、各要素の重みを線形補間などで滑らかに変化させます。
- **序盤（空きマス 60〜40）**: Mobilityを最重視。石の数は少なく保つ（Evaporation戦略）。
- **中盤（空きマス 40〜15）**: MobilityとCorner、Stabilityをバランスよく評価。
- **終盤（空きマス 15〜0）**: 完全読み（Perfect Search）に切り替え、最終的な石の差（Coin Parity）のみを最大化します。

## 4. 終盤の完全読み（Endgame Solver）

オセロの終盤（残り10〜15マス程度）は、評価関数を使わずに最後まで読み切る「完全読み」が可能です。

- **Win/Loss/Draw探索**: まずは勝敗のみを高速に判定する探索を行います。
- **Exact Score探索**: 勝敗が確定した後、最終的な石の差を最大化（または最小化）する探索を行います。
- 終盤は枝刈りが非常に効きやすいため、ビットボードと組み合わせることで、残り15マス程度であれば150ms以内に完全に読み切ることが可能です。

## 5. CodinGame特有の対応

- **Expert Modeの活用**: 相手の行動履歴を受け取る機能ですが、自前で盤面を管理・更新する本実装においては、標準の盤面入力（文字列）から状態を構築するだけで十分なため、必須ではありません。ただし、デバッグ用途として活用できます。
- **時間管理**: `std::chrono` を用いて厳密に時間を計測し、150msの制限時間を超過しないように安全マージン（例: 140msで打ち切り）を設けます。

## 6. 開発ステップ

1. **基盤実装**: ビットボードによる盤面表現、合法手生成、石の反転処理の実装とテスト。
2. **探索アルゴリズム**: NegaScout法と反復深化の実装。
3. **評価関数**: Mobility, Corner, Stabilityを考慮した評価関数の実装とチューニング。
4. **高速化**: 置換表（Zobrist Hashing）とMove Orderingの導入。
5. **終盤ソルバー**: 残りマス数が少ない場合の完全読み処理の実装。
6. **CodinGame連携**: 入出力処理の実装と、プラットフォーム上でのテスト・調整。
